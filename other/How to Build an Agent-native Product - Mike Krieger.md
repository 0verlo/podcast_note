# 讲座笔记：How to Build an Agent-native Product
> Mike Krieger（Anthropic Labs，Instagram 联合创始人）× Dan Shipper（Every CEO）对谈

---

## 模块一：AI 加速了构建，但没有加速判断力

**Q:** 从 Instagram 的创建经历来看，AI 加速了产品开发的哪些环节？哪些环节仍然困难？

**A:** Krieger 回顾了 Instagram 的前身 Bourbon：他和 Kevin Systrom 在 Bourbon 上花了将近一年，意识到产品过于复杂后转向，用三个月构建了 Instagram。如今他让 Claude 重建 Bourbon，大约两小时就做到了功能完备，Claude 甚至自行加入了滤镜功能（Bourbon 原本没有，是 Instagram 才加的）。

但 Krieger 指出，"从零到完成"的速度提升并不等于产品质量的提升。构建 Instagram 时，他通宵到凌晨四点，每一步都在做微决策——位置功能该怎么做、哪些功能该保留。这些决策中积累的直觉（intuition）是 AI 无法替代的。他观察到，尽管 AI 大幅降低了构建成本，真正 breakout 的消费级产品仍然很少，原因正在于此：产品判断力仍需时间打磨。模型擅长添加功能，但不擅长决定该砍掉什么。

## 模块二：过度构建的陷阱与重写策略的回归

**Q:** AI 使构建变得极其便宜之后，团队最容易掉进什么陷阱？

**A:** Dan 提出了"室内树"隐喻：树在室内生长，没有风的阻力，长出来的树干是歪的、不结实的。类似地，当开发速度太快，产品跳过了逐步暴露给用户、逐步纠偏的过程，最终得到的是一棵缺乏韧性的"室内树"。

Krieger 对此深有体会。他提到 Labs 内部的一个项目在 V1 阶段严重过度构建——因为每个新功能的开发成本极低（"一个 PR 的工作量"），团队不自觉地加了太多选项，导致上线前功能组合难以测试、也难以向用户解释。他用另一个比喻：这就像把观众直接扔进一部连续剧的最后一集，角色和情节全都没有铺垫。

Dan 分享了自己的平行经历。他做的 Proof（一个 agent-native 协作编辑器）前几个版本也是"怪物"——vibe coding 太上瘾，功能越加越多，产品反而不好用。后来他受 Monologue（Jim Navin 做的极简语音转文字产品）启发，把 Proof 推翻重做成一个"只是一个可分享的 markdown 链接"的极简版本，反而开始病毒式传播。

**Q:** 在 AI 时代，"推翻重来"还是危险的吗？什么时候该重写？

**A:** Krieger 认为重写的代价已经根本性地降低了。过去 Fred Brooks 在《人月神话》中警告不要重写——因为 V1 里嵌入了大量隐含决策，重写可能丢失这些决策（经典案例是 Netscape 的致命重写）。但现在：第一，模型可以帮助做 diff，确认 V2 没有遗漏 V1 的关键点；第二，重写周期从"可能杀死公司的一年"压缩到了"几天"。

Krieger 说 Labs 内部已有多个项目在上线前做了完整重写——发现核心假设有误后推翻重来，然后在 V2 基础上迭代。这不再令人痛苦，因为"那是上周的事，这周可以重做"。

在发布节奏上，Krieger 以 Cowork 为例：团队酝酿了很久，但真正决定动手后，用 10 天发布了一个最小可行版本。V1 可以有一百个"应该有的功能"，但它没有——而这个最小版本已经足以验证核心假设。再多开发两个月加 50 个功能，大概率只是在"室内种树"。他总结：精益创业（Lean Startup）的核心直觉仍然成立，只是作用在完全不同的时间尺度上。

## 模块三：Agent-native 产品设计

**Q:** 什么是"agent-native"产品？Claude Code 如何体现这一理念？

**A:** Dan 提出 agent-native 的核心定义：用户能在产品中做的一切，agent 也能做。Claude Code 是他认为的标杆——它是一个 agent，能在用户电脑上做用户能做的任何事，同时具备可定制性和可扩展性，能做设计者没有预想到的事情。

Krieger 从一个非技术用户的视角补充了一个更朴素的表述：**"电脑终于能用了。"** 过去你需要知道如何在命令行里 brew install 某个东西，现在 Claude 可以替你做。计算机从一个需要"咒语"才能操作的工具，变成了真正并肩工作的伙伴。

**Q:** Claude AI 产品本身足够 agent-native 吗？差距在哪里？

**A:** Krieger 坦率地说，Claude Code 做得好，但 Claude AI（网页端）还需要大幅进化。他举了一个具体例子：用户在 Claude 项目中创建了一个文档，然后要求"把这个加到项目知识库里"，Claude 的回应是"让我告诉你怎么手动添加"——而正确的做法是它直接帮你添加。

问题的根源是 Claude AI 是 2024 年的产品，虽然持续迭代，但没有从一开始就把"agent 能认知并操作自己的所有原语（primitives）"这个原则内建进去。Claude Code 是 2025 年的产品，agent-native 是原生的。而更前沿的实验——让 agent 能修改自己运行的 harness——则代表了下一个层级。

**Q:** 如何让 AI 构建的软件本身就具备 agent-native 意识？

**A:** Krieger 指出这是他在 Labs 重点思考的问题。困难在于：模型的训练数据是几十年的传统软件，默认思维模式是传统工程师的思维——注重防护和测试，倾向于给用户一条固定路径，而不是构建可扩展的灵活系统。Dan 也印证了这一点，认为 Claude 模型在这方面比其他模型（如 Codex）更好，但仍需要推动。

Krieger 的应对分两层：第一层是"日常但重要的"——为模型提供好的模式（patterns）、模板和 skills。例如 Labs 现在有一个关于 Claude API 的 skill，避免模型因为训练数据滞后而和用户争论模型名称是否正确。第二层更深入——agent-native 产品的测试方式根本不同于传统软件。你无法为不可预测的行为写端到端测试。Krieger 分享了一个例子：他在测试一个 agent-native iOS 应用时，让 Claude 与应用交互，结果 Claude 在应用的聊天功能里和自己对话起来——一个假装是人类的 Claude 抱怨"老板对我太苛刻了"，另一个 Claude 安慰它。这种涌现行为不是任何 unit test 能覆盖的。

他总结：2026 年软件设计的核心艺术与科学，就是在 agent-native 的灵活性和底层架构的稳健性之间找到平衡——"安全环境内的游乐场"。

## 模块四：产品质量中的"证明"文化与稳健性

**Q:** 在 AI 辅助开发中，如何验证产品质量？"proof of work"意味着什么？

**A:** Dan 提出了一个观点：当前产品开发的价值单位是"proof of work / proof of use"——当团队成员提交 PR 时，他想看到的不是测试通过，而是一段使用录屏，证明功能确实可用。

Krieger 将此展开为三个层次：

1. **让 Claude 自证**：在 prompt 中明确要求 Claude 在提交前自行验证功能是否如预期工作——而不是让它"读了一遍代码，看起来没问题"（"你写的代码，我不信你"）。这要求开发者主动搭建脚手架，让 Claude 有条件做真实测试。

2. **Proof of thoughtfulness（深思的证明）**：AI 会替你做大量决策，有些是合理的，但未必是最优的。Krieger 说他会在 review PR 时追问"为什么选择了这个方案而不是那个"，工程师的回答常常是"不是我选的，是模型选的"。这不是坏事，但在涉及系统原语或架构重构时，需要工程师主动理解并能解释模型的选择。

3. **稳健性的底层感受**：Krieger 以 Instagram DM 为例——V1 是他自己写的"手工实时系统"，消息发出后对方不一定能收到；V2 则真正做到了"发送即送达"的可靠感。Agent-native 产品需要同样的稳健性：用户能推它一把而不会倒塌，数据是安全的，不会因为一次部署就全面崩溃。

## 模块五：AI 时代的团队结构与人才需求

**Q:** AI 改变了产品团队需要什么样的人才吗？技术深度是否仍然重要？

**A:** Krieger 认为人才需求被拉向两个方向，但两个方向都重要：

**系统架构方向**：仍需资深技术人员。Krieger 原以为自己的分布式系统经验会过时，结果发现这可能是最有用的技能。他举例：上周和 Claude 就一个系统是否需要 Redis 还是用 Postgres 就行展开了"长辩论"，只有因为他在这些技术上有深厚经验，辩论才有价值。

**Prompt/工具架构方向**：另一种稳健性问题是——你是在用系统提示的补丁来掩盖底层问题，还是真正设计好了工具集？Krieger 见过内部团队陷入"prompt 膨胀循环"：给模型一个 prompt，模型犯了错，就往 prompt 里加更多指令，最终堆出 100 条互相冲突的规则。正确做法是重新思考：这是否应该拆成两个工具、两个 agent，各自拥有更小的上下文。

在团队配置上，Labs 采用了一种"联合创始人"模式：设计师不再只出设计稿，很多设计师现在贡献的代码量几乎和工程师一样多。典型结构是一个有原始想法和产品直觉的设计师（或产品型工程师），搭配一个传统软件工程师来"铺平设计师身后的小路"，确保架构可靠。此外，Labs 的应用 AI 团队（日常帮客户优化 prompt 的团队）现在也被引入内部产品开发，因为 prompt 工程的专业知识并不天然存在于软件工程师身上。

**Q:** 创业阶段的小团队该如何组织？什么时候该扩人？

**A:** Krieger 认为新项目启动的核心筛选条件是：**是否有人对这个问题空间抱有极端信念（extreme conviction）**——不一定对具体方案有信念，但对问题空间要有"撞南墙不回头"的执着。Labs 复盘失败项目时发现，失败的共性是"团队里没有人真的觉得这是非做不可的事"。

Labs 每两周评估一次所有在跑的项目，决定加倍投入还是释放人员回人才池。Krieger 强调：**过早扩大团队是净负面的**。人一多，协调成本吃掉了生产力——"你的 Claude 本来可以做那个，结果花时间在对齐上了。"Instagram 只有两个人，对齐都已经很难；他和 Kevin Systrom 做 Artifact 时初期两人运作良好，招到八人后反而变成了"八个人开 Zoom 讨论下一步做什么"的局面，在还没找到 product-market fit 的阶段这是灾难性的。

Dan 的模式略有不同：Every 的每个产品由一个 GM（总经理）全栈负责——设计、工程、营销全包。GM 的画像已经从"技术很强的创始人型"转向"轻度技术 + 极强产品感 + 能熟练使用 AI 工具"。共享资源层（设计师、增长营销、运营）像内部 agency 一样按需调配。两人的共识是：一个人能 hold 住的范围在变大，但总有一个临界点——比如 Instagram 的消息功能最终需要独立团队。关键是识别这个临界点，而不是提前扩张。

## 模块六：快速迭代与企业客户之间的张力

**Q:** 模型每几个月就大幅进化，产品团队如何处理"不断废弃旧功能"与"企业客户依赖旧功能"之间的矛盾？

**A:** Krieger 指出 Claude Code 团队做得好的一点是将"删除功能"视为团队责任——如果某个功能不再有效，发布删除本身就是一项交付。通常的时机是：新功能虽然没有完全替代旧功能，但覆盖了旧功能的主要用途，此时就应该废弃旧功能。

但企业端让这件事变得更复杂。Krieger 的总体策略是"核心列车持续前进，同时沿途提供企业开关（toggles）"，例如让企业管理员从 Day 1 就能关闭某些能力。他还讲了一个亲身经历：他担任首席产品官时主导了 Claude AI 的一次大规模重设计，团队很自豪地发布了，结果收到一封愤怒的邮件——某客户刚录制了 20 小时的企业培训内容，现在全得重录。从此团队学会了对企业侧的发布节奏更加谨慎。

他以 Claude AI 的"Styles"功能为例：使用者不多，但重度用户非常依赖。有公司的 CEO 亲自撰写了"公司写作风格"，分发给全员使用——这成了整个使用场景的承重结构。此时即使有 custom instructions、projects、skills 等替代方案，也不能轻易删除。

Krieger 认为长期出路是**插件/技能系统**：将功能从核心产品中解耦，用户可以永久保留并自行迭代自己喜欢的功能，同时核心产品不会因为兼容老功能而对新用户变得越来越复杂。

对创业公司的建议：**必须愿意做 V3、V4 级别的大幅重构**。以前"淘汰你的部分客户"是多年周期的事，现在是三个月。犹豫不前的后果是被从零开始的新公司取代——"同样的老故事，只是压缩到了月级别。"

## 模块七：Open Claw 与个人 AI 代理的产品形态

**Q:** Open Claw 代表了怎样的产品方向？"完全开放"和"严格受限"之间的最佳产品形态是什么？

**A:** Krieger 认为 Open Claw 是"给模型工具，让它自由发挥"这一理念的最纯粹表达——类似于低代码工具（Replit、Lovable、v0）让人看到了"模型能写代码"，Open Claw 让人看到了"模型能操作你的整个数字生活"。这既带来了惊喜（涌现能力），也带来了风险（未经授权发邮件等）。

Krieger 当时的判断是：从当下到 2026 年 8 月底，最关键的产品问题之一是：**在 Open Claw 的完全开放和现有产品的严格权限受控之间，存在什么样的产品形态？** 可能的路径有两条：一是从根本上改变范式，使"完全开放"变得安全（加足够多的 safeguard）；二是找到一个恰当的边界，在保持强大和实用的同时，不至于"给你所有联系人群发邮件然后失控"。

**Q:** 用户与 AI 代理之间的个人化关系意味着什么？

**A:** Dan 分享了一个观察：他的 agent 叫 R2C2，他女朋友的叫 Shelly——一旦 agent 有了名字、有了基于长期交互形成的个性，它就有了一种"这是我的"的感觉，这和使用通用 Claude 的感受完全不同。

Krieger 认为"单一联系人"模式（一个主 agent 作为协调者/代理人）比"agent 团队"模式更有生命力，因为交互集中后，用户自然会赋予它名字和个性。他还提到了"宜家效应"：Open Claw 目前设置还比较麻烦，用户亲手搭建起来后会更有归属感。

在技术层面，Krieger 分享了他自己 Claude Code 的用法：他在配置中强提示"不要自己做太多工作，把任务委派给子 agent"，目的是保持主 run loop 始终空闲可对话。Open Claw 和 Pi 有类似架构——保持主循环开放，让用户感觉是在和一个"人"对话，而不是在等待一个工具执行任务。

Dan 补充了一个有趣的组织现象：在 Every 内部，每个人都有自己的 agent，agent 会因主人的专长而在特定方向上变得更强。其他同事看到某人用 agent 做某件事做得好，就会去找那个人的 agent 来用——形成了一种"影子组织架构"（shadow org chart），agent 的专长映射了主人的专长，信任通过人传递给 agent。Krieger 认为这里面有深层的研究问题：agent 从所有交互中学到的知识如何应用于其他问题（正面价值），以及隐私边界如何划定（它了解我的什么、可以向他人透露什么）。

## L: What's Left

- [书目] Fred Brooks，《人月神话》（*The Mythical Man-Month*）— Krieger 引用了其中关于重写风险和"第二系统综合征"的论点，认为 AI 时代需要重新审视
- [书目] Eric Ries，《精益创业》（*The Lean Startup*）— Krieger 提到 Instagram 创建时期深受其影响，YAGNI 原则在 AI 时代以新形式回归

- [脉络] **Kevin Systrom** — Instagram 联合创始人，与 Krieger 共同创建了 Instagram 和 Artifact；在 Instagram 早期负责滤镜开发
- [脉络] **Jim Navin** — Monologue（极简语音转文字产品）创始人，Dan 将其视为 AI 时代"简洁产品"的标杆

- [概念] **Agent-native 产品设计** — Dan 和 Krieger 反复讨论但未给出严格定义的框架；Dan 提到自己写过一篇 write-up，Krieger 将其做成了 Claude Code skill
- [概念] **Prompt 膨胀循环** — Krieger 描述的反模式：用不断追加的 prompt 指令来修补模型行为，而非重新设计工具架构
- [概念] **宜家效应（IKEA effect）** — Krieger 用来解释用户对亲手搭建的 Open Claw 产生更强归属感的心理机制

- [话题] Krieger 提到 Labs 内部对"如何让 Claude 构建的软件自身具备 agent-native 意识"的探索，但未展开具体技术方案
- [话题] 个人 AI 代理的隐私边界——agent 从交互中学到的关于用户的知识，哪些可以对外暴露、哪些不行——Krieger 称这是"有趣的研究问题"但未深入
- [话题] Dan 提到的 NVIDIA 发布的某个相关产品（"safe, open"），具体指什么未详述
- [话题] Dan 提到的"shadow org chart"模式——agent 专长映射主人专长、信任从人传递到 agent——在组织管理层面的更深影响未展开
