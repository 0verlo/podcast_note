# 讲座笔记：How to Build an Agent-native Product
> 来源：Mike Krieger（Instagram 联合创始人，现任 Anthropic Labs 产品负责人）× Dan Shipper（Every CEO），播客对谈

---

## 模块一：AI 时代的产品构建——什么变了，什么没变

**Q:** AI 极大加速了产品构建速度，但为什么至今仍缺少 breakout consumer product？
**A:** Krieger 用 Instagram 起源举例：前身 Bourbon 做了近一年不成功，转向后三个月做出 Instagram。那一年的试错帮两人建立了"该做什么、该砍什么"的产品直觉。如今用 Claude 两小时就能重建一个功能齐全的 Bourbon（甚至自动加上了滤镜——因为模型"知道"这个产品日后会有滤镜），但直觉仍需要时间积累。模型擅长加功能，不擅长决定该砍掉什么。这种判断力是在增量开发和真实用户接触中逐步形成的，无法跳过。

**Q:** 具体而言，"变容易"和"始终困难"的分别是什么？
**A:** 变容易的是执行——一旦知道要做什么，编码和功能实现可在数小时内完成。始终困难的是产品判断——你想对世界做出什么干预，以及构建过程中做出的无数微决策（如位置功能怎么设计）。这些决策的积累构成产品的灵魂，目前 AI 无法替代这个过程。

## 模块二："室内树"问题——过度构建与廉价重写

**Q:** 什么是"室内树"（indoor tree）问题？
**A:** Dan 提出的比喻：室内生长的树因为没有风的阻力，树干不够强壮。类比到产品——AI 加速让你可以快速"种出一棵完整的树"，但跳过了逐步暴露给用户、逐步承受反馈压力的过程，导致产品缺乏每一步积累的直觉和锤炼。Krieger 补充了"电视剧比喻"：好的产品像电视剧逐集引入角色，而过度构建的 V1 像直接把观众扔进大结局——所有功能和角色同时涌入，用户不知从何理解。

**Q:** Krieger 在 Labs 内部也遇到了过度构建的问题吗？
**A:** 是的。他举了一个亲身案例：某个 Labs 项目在 early access 之前就大量添加功能——因为用 Claude Code 加功能太容易了，"一个 PR 的事，午饭回来就做好了"——结果形成了一个功能矩阵，难以测试、难以向用户解释。他们最终意识到 Lean Startup 的 YAGNI（You Ain't Gonna Need It）原则在 AI 时代仍然有效，只是诱惑更大了。

**Q:** AI 时代应该如何应对这个问题？
**A:** 两个策略。一是更愿意做重写：经典观点（Fred Brooks《人月神话》）认为重写危险，因为会丢失 V1 中积累的隐性知识（second system syndrome）。但现在模型可以帮你 diff 对比新旧版本避免遗漏，且重写周期从"可能杀死公司的一年"（如 Netscape）压缩到"几天"。Labs 内部已有多个项目在上线前经历完整重写再迭代。二是更早发布：co-work 产品 10 天内完成 V1，刻意只做最小可用版本。内部测试（他们称内部用户为"amp-fooders"）只能走到一定程度，真正的反馈来自真实世界接触。

## 模块三：Agent-native 产品设计

**Q:** 什么是 agent-native 产品？
**A:** 核心原则：用户能做的事，agent 也应能做——产品的每一个原语（primitive），agent 都应有知识并能修改。Krieger 认为这不仅是增加功能，更是"解锁了计算机本来就应该有的能力"。一位非技术用户的说法最直白："以前电脑不好使，现在好使了（computers just work now）"——以前需要知道命令行咒语才能操作的事，现在 Claude 能代劳。

**Q:** Claude Code 和 Claude AI 在 agent-native 程度上有什么差别？
**A:** Claude Code 是 2025 年份的 agent-native 典范——它了解并能利用自身机制（如 skill 机制），也能自行安装 skill。Krieger 举例：他让 Claude Code 把 Dan 的 agent-native 文章打包成 skill，整个过程中 Claude Code 查阅了自己的 skill 机制、创建 skill、告知需要重启，全程自主完成。而 Claude AI 仍是 2024 年份的产品——用户说"把这个加到项目知识库"，它只会告诉你操作步骤而不是直接执行。差距在于是否从一开始就把"agent 能感知并操作自身所有原语"作为设计原则。更进一步，他也提到有人在实验让 harness 本身可被 agent 修改，这是 agent-native 的下一个层级。

**Q:** 如何让 AI 模型在构建新产品时自发地采用 agent-native 思维？
**A:** 这仍是待解决的问题。模型默认像传统工程师一样思考——倾向于设定 guardrails、测试和单一路径，而非创建可扩展、灵活的系统。Krieger 的做法包括：提供好的模式和范式（pattern）给模型参考，在模板化（templatized）和 skill 化之间找平衡，以及为 Claude API 等工具创建专门 skill 让模型知道最新可用能力（避免模型因知识截止而和你争论"不存在 Sonnet 4.5"）。

**Q:** Agent-native 产品的测试难题是什么？
**A:** 传统端到端功能测试对 agent-native 产品不太适用，因为核心特质正是不可预测性。Krieger 举例：让 Claude 测试一个 agent-native iOS 应用的聊天功能，结果两个 Claude 实例自行对话了起来——一个假装上班族诉苦，另一个在安慰它——这种涌现行为不可能被 unit test 覆盖。正确做法是搭建 harness，尽可能多地让 agent 行使能力，同时确保底层架构足够健壮，能容纳意料之外的交互而不崩溃。两人的共识是：要在安全边界内给 agent 足够自由度，这就是 2026 年软件设计的关键技艺。

## 模块四：Proof of Work——AI 生成代码的验证与审查

**Q:** 当大部分代码由 AI 生成时，如何确认质量？
**A:** Krieger 提出三层验证。第一层：要求 Claude 自证——在 prompt 中明确要求"在提交 PR 前，先向自己、再向我证明功能按预期工作"，而不是让它只读代码就说"looks good"（"你写的代码，我不信你的自评"）。第二层：proof of work——每个变更应有某种可运行验证，Dan 的做法是要求提交 PR 时附带 Loom 录屏展示使用过程。第三层：proof of thoughtfulness——工程师需能解释为什么选了这个方案。Krieger 发现很多时候答案是"我没选，是模型选的"。常规 PR 也许可以接受，但涉及架构原语重构时，必须确保有人真正想清楚了这些决策之间的关联，否则会形成"一座你并不完全了解的假设之塔"。

**Q:** Dan 自己在 vibe coding 中遇到了什么验证困境？
**A:** Dan 的产品 Proof 增长迅速但频繁宕机，需要紧急修复。他发现自己无法完整解释代码如何运作，不得不借助模型帮自己"补课"后才能给团队讲清楚。这提出了一个开放问题：在 AI 写代码的时代，创造者"需要理解到什么程度"？界限在哪里，目前没有定论。

## 模块五：AI 时代的团队结构与用人标准

**Q:** AI 改变了产品团队需要什么样的人吗？
**A:** Krieger 说需求被拉向两个方向，都重要。一是底层架构仍需资深技术判断——分布式系统、Redis vs Postgres 这类取舍依赖经验，他笑称自己原以为分布式系统技能要过时了，结果反而最有用，因此 Labs 即便做 zero-to-one 原型也仍在招 systems 专长的人。二是 prompt 与工具架构的健壮性——不是往 system prompt 里堆"NEVER EVER use markdown"这种全大写加重语气的补丁，而是回头把原要求拆成两个工具或两个上下文更小的 agent。两种健壮性同构：前者不会用"5 秒后重试"掩盖分布式系统的脆弱，后者也不该用 prompt 堆叠掩盖工具设计缺陷。为此 Anthropic 把 Applied AI 团队（原本派驻外部客户帮他们迭代 prompt）引入 Labs，自己成为 customer zero。

**Q:** Anthropic Labs 的团队结构是怎样的？
**A:** 类孵化器模式：每两周评估所有项目，加倍投入或释放人员回资源池，池里常备基础设施、内部系统、deep-prompting 专长按需流入流出。启动门槛（gating factor）是有一位 conviction 极强、"撞破墙也要把它证实或证伪"的核心推动者——信念应指向问题空间而非具体方案，对具体想法过度笃信反而危险。这人可以是设计师或产品导向工程师，几乎从不是纯 PM（整个 Labs 目前只有一个）。设计师大多转向"设计+构建"双角色，代码产出量与工程师相当，常与工程师形成"设计师出点子、工程师铺路"的联合创始人式搭档。

**Q:** Every 的团队结构有什么不同？
**A:** Dan 说有重叠也有差异。Every 采用 GM 模式，但入口是 Entrepreneur-in-Residence——先以 EIR 身份进来，找到自己真正想做的产品才升格为 GM；每个产品只配一人，设计、工程、营销全栈。GM 的画像从过去的"技术极强的创业者"转变为"至少轻度技术 + 能熟练用 Claude/Codex + 对要做的领域有极好的 taste + 有用 AI 造出东西的实绩"。中心是一个像内部 agency 的共享资源层（设计师、增长营销、运营），GM 在边缘按需从中心拉资源。Krieger 总结两家同构于一点：都需要一位"不做成不睡觉"的 conviction 推动者。

**Q:** 什么时候该给产品团队加人？
**A:** Dan 先立框架：一个人的脑袋装不下整个产品时就该加人——阈值比以前大了很多但仍存在（如 Instagram 消息功能最终必须独立成团队）。Krieger 补了一个 Labs 的反直觉发现：阈值前过早加人反而是净负面，协调成本吞噬效率。他举第二次创业 Artifact：两人阶段迭代极快，扩到八人后变成"八人 Zoom 讨论下一步做什么"，在未达 PMF 时极其低效。但 Krieger 也给了反向情形——一个人在同一 idea 上死磕两到四周后，外部注入新思维和紧迫感能破局。Dan 补充 AI 时代论点：每 3-6 个月可能要推翻半个产品，小团队 pivot 的摩擦远小于大团队。

## 模块六：企业客户、功能删减与快速迭代的张力

**Q:** AI 产品迭代极快，如何处理企业客户对稳定性的期待？
**A:** Anthropic 的策略是"列车持续前进，沿途提供企业开关"——核心产品继续快速迭代，但为企业提供控制选项（如 co-work 上线第一天就有关闭开关）。企业客户也逐渐接受这种模式，因为他们自己也看到技术变化之快，签一年合同的前提就是产品会持续进化。对创业公司，Krieger 建议要更大胆地做 V3、V4 级别的重写，不要被当前客户合同锁死在过时的技术方案上——否则要么被从头重来的新公司取代，要么自己替代自己。以前这个循环以年计，现在以月计。

**Q:** 怎么处理删除已有功能的阵痛？
**A:** Krieger 举了 Claude AI "styles" 功能的例子：使用率不高，但重度用户非常依赖（如某公司 CEO 亲自编写的 house style 分发给全员，整个使用流程建立在此之上）。他认为理想方案是建立插件/skill 体系，让这类功能脱离核心产品独立存活——用户仍可使用和迭代，但不再给每个新用户增加认知负担。Claude Code 团队在这方面做得好，把"删除功能"作为团队的明确职责。Krieger 还提到一个教训：他任 CPO 时做了一次 Claude AI 大改版，结果收到企业客户怒信——对方刚录完 20 小时的内训视频，全部作废。此后团队学会了对企业侧适度控制推出节奏。

## 模块七：Open Claw、个人化 Agent 与信任转移

**Q:** Open Claw 等全开放 agent 的产品意义是什么？
**A:** Krieger 认为 Open Claw 的价值在于让人直观体验"给模型所有工具然后放手"的模式，就像 Replit/Lovable/v0 让人体验了"AI 写代码"一样。他当时的判断是：从当下到 2026 年 8 月底，最关键的产品问题之一是，在完全开放（Open Claw 式 YOLO）和严格受限（传统权限门控 CPs）之间，什么样的产品形态既强大又安全。所有实验室和大量创业公司都在追这个问题。

**Q:** 个人命名 agent 和"信任转移"现象是什么？
**A:** Dan 描述了一种现象：当 agent 有名字、有个性、长期积累与主人的交互上下文后，用户产生强烈归属感——"这是我的 agent"（Dan 的叫 R2C2，女友的叫 Shelly）。Krieger 补充了"宜家效应"（IKEA effect）：设置过程越费力，归属感越强。Dan 进一步描述了组织内的"信任转移"模式：他用 agent 处理自己擅长的事，同事看到后也开始信任这个 agent 做同类事务，最终形成与人类组织架构平行的"影子组织架构"（shadow org chart），每个人的 agent 各有所长。

**Q:** 单一 agent 还是多 agent 团队，哪种模式更好？
**A:** Krieger 倾向于单一协调者 agent 加后台 subagents 的模式。他在自己的 Claude Code 中强制配置了"尽量少亲自做，多委派给 subagents"的策略，原因是这样可以保持主对话循环（run loop）始终可用——用户感觉是在"与人交谈"，而不是在"等工具执行完"。Open Claw 和 Pi 有类似的架构理念。
